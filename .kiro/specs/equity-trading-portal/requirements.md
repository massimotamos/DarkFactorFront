# Requirements Document

## Introduction

The Equity Trading and Investment Portal is a complete replacement of the existing Angular + Spring Boot platform. It serves retail traders and investors by collecting their financial profile, investment time horizon, regional preferences, and target return on investment, then generating a tailored multi-asset portfolio recommendation. The portal also proposes derivative overlays (vanilla options, swaps, exotic options) to hedge or manage risk over the selected horizon. Market data is sourced from free-of-charge providers, with real-time feeds preferred and periodic polling as a fallback.

The frontend is rebuilt in Angular (TypeScript) and the backend is rebuilt as a Java Spring Boot multi-module Maven project, both containerised via Docker.

## Glossary

- **Portal**: The complete equity trading and investment web application (frontend + backend).
- **User**: A retail trader or investor interacting with the Portal.
- **Profile_Service**: The backend service responsible for collecting and persisting User financial profiles.
- **Portfolio_Engine**: The backend service responsible for generating tailored portfolio recommendations.
- **Risk_Engine**: The backend service responsible for proposing derivative overlays to manage portfolio risk.
- **Market_Data_Service**: The backend service responsible for fetching, caching, and distributing market data from external providers.
- **Auth_Service**: The backend service responsible for User authentication and session management.
- **Instrument**: A tradeable financial asset supported by the Portal (equity, bond, vanilla option, swap, exotic option, or structured product).
- **Equity**: A share representing ownership in a publicly listed company.
- **Bond**: A fixed-income debt instrument issued by a government or corporation.
- **Vanilla_Option**: A standard European or American call or put option on an underlying asset.
- **Swap**: An OTC derivative contract exchanging cash flows between two parties (e.g. interest rate swap, equity swap).
- **Exotic_Option**: A non-standard option with path-dependent or complex payoff structures (e.g. barrier, Asian, lookback).
- **Structured_Product**: A pre-packaged investment combining a fixed-income component with one or more derivative overlays.
- **Financial_Profile**: A record capturing a User's risk tolerance, investment experience, income bracket, and net worth band.
- **Investment_Horizon**: The duration (in months or years) over which a User intends to hold a portfolio.
- **Regional_Preference**: A User-specified geographic focus for investments (e.g. North America, Europe, Asia-Pacific, Global).
- **Target_ROI**: The annualised percentage return a User wishes to achieve.
- **Portfolio_Recommendation**: A weighted allocation across Instruments generated by the Portfolio_Engine for a specific User.
- **Derivative_Overlay**: A set of derivative positions proposed by the Risk_Engine to hedge or enhance a Portfolio_Recommendation.
- **Market_Data_Provider**: An external free-of-charge data source supplying price quotes, historical data, and reference data (primary: Yahoo Finance via yfinance-compatible API; fallback: Alpha Vantage free tier).
- **Data_Cache**: An in-memory or persistent store holding the most recent market data snapshot to reduce external API calls.
- **WebSocket_Feed**: A persistent bidirectional connection delivering streaming market data updates to the frontend.
- **REST_API**: The HTTP/JSON interface exposed by the backend to the frontend.
- **JWT**: A JSON Web Token used for stateless authentication between frontend and backend.

## Requirements

### Requirement 1: User Registration and Authentication

**User Story:** As a retail investor, I want to register an account and log in securely, so that my profile and portfolio data are protected and personalised.

#### Acceptance Criteria

1. WHEN a User submits a registration request with a unique email address and a password of at least 12 characters, THE Auth_Service SHALL create a new User account and return a success response within 3 seconds.
2. IF a User submits a registration request with an email address already associated with an existing account, THEN THE Auth_Service SHALL return an error response with HTTP status 409 and a descriptive message.
3. IF a User submits a registration request with a password shorter than 12 characters, THEN THE Auth_Service SHALL return an error response with HTTP status 400 and a descriptive message.
4. WHEN a User submits valid credentials, THE Auth_Service SHALL return a signed JWT with an expiry of 60 minutes and a refresh token with an expiry of 7 days.
5. WHEN a JWT expires, THE Auth_Service SHALL accept a valid refresh token and issue a new JWT without requiring the User to re-enter credentials.
6. IF a User submits invalid credentials, THEN THE Auth_Service SHALL return an error response with HTTP status 401 and SHALL NOT reveal whether the email or password was incorrect.
7. WHEN a User requests logout, THE Auth_Service SHALL invalidate the refresh token so that it cannot be used to obtain new JWTs.
8. THE Auth_Service SHALL store passwords as salted hashes using bcrypt with a cost factor of at least 12.

### Requirement 2: Financial Profile Collection

**User Story:** As a retail investor, I want to provide my financial profile, investment horizon, regional preferences, and target ROI, so that the portal can generate a portfolio recommendation suited to my situation.

#### Acceptance Criteria

1. WHEN an authenticated User submits a Financial_Profile containing risk tolerance (one of: conservative, moderate, aggressive), investment experience (one of: beginner, intermediate, advanced), income bracket, net worth band, Investment_Horizon (1–360 months), at least one Regional_Preference, and a Target_ROI (0.1%–50% per annum), THE Profile_Service SHALL persist the Financial_Profile and return HTTP status 201.
2. IF an authenticated User submits a Financial_Profile with an Investment_Horizon outside the range 1–360 months, THEN THE Profile_Service SHALL return HTTP status 400 with a descriptive validation error.
3. IF an authenticated User submits a Financial_Profile with a Target_ROI outside the range 0.1%–50% per annum, THEN THE Profile_Service SHALL return HTTP status 400 with a descriptive validation error.
4. IF an authenticated User submits a Financial_Profile with no Regional_Preference selected, THEN THE Profile_Service SHALL return HTTP status 400 with a descriptive validation error.
5. WHEN an authenticated User updates an existing Financial_Profile, THE Profile_Service SHALL replace the previous record and return HTTP status 200.
6. WHEN an authenticated User retrieves their Financial_Profile, THE Profile_Service SHALL return the most recently persisted Financial_Profile for that User.
7. THE Profile_Service SHALL validate that all required fields are present before persisting a Financial_Profile.

### Requirement 3: Portfolio Recommendation Generation

**User Story:** As a retail investor, I want to receive a tailored portfolio recommendation based on my financial profile, so that I can make informed investment decisions aligned with my goals.

#### Acceptance Criteria

1. WHEN an authenticated User requests a Portfolio_Recommendation and a valid Financial_Profile exists for that User, THE Portfolio_Engine SHALL generate and return a Portfolio_Recommendation within 10 seconds.
2. THE Portfolio_Engine SHALL include at least one Instrument from each of the following asset classes present in the User's Regional_Preference: Equity, Bond.
3. WHERE a User's risk tolerance is set to aggressive and the Investment_Horizon is greater than 36 months, THE Portfolio_Engine SHALL include at least one Structured_Product in the Portfolio_Recommendation.
4. THE Portfolio_Engine SHALL ensure that the sum of all allocation weights in a Portfolio_Recommendation equals 100%.
5. THE Portfolio_Engine SHALL ensure that no single Instrument allocation exceeds 40% of the total Portfolio_Recommendation weight.
6. IF an authenticated User requests a Portfolio_Recommendation and no Financial_Profile exists for that User, THEN THE Portfolio_Engine SHALL return HTTP status 422 with a message instructing the User to complete their Financial_Profile.
7. WHEN a Portfolio_Recommendation is generated, THE Portfolio_Engine SHALL persist the recommendation with a timestamp so that the User can retrieve historical recommendations.
8. WHEN an authenticated User retrieves historical Portfolio_Recommendations, THE Portfolio_Engine SHALL return all persisted recommendations for that User ordered by timestamp descending.
9. THE Portfolio_Engine SHALL use current market prices from the Market_Data_Service when computing expected returns for each Instrument in the Portfolio_Recommendation.

### Requirement 4: Derivative Overlay and Risk Management

**User Story:** As a retail investor, I want the portal to propose derivative strategies to hedge or manage risk on my portfolio, so that I can protect my investments over my chosen time horizon.

#### Acceptance Criteria

1. WHEN a Portfolio_Recommendation is generated, THE Risk_Engine SHALL produce a Derivative_Overlay containing at least one proposed derivative position for the portfolio.
2. THE Risk_Engine SHALL propose Vanilla_Option positions (protective puts or covered calls) for Equity holdings in the Portfolio_Recommendation where the Equity allocation exceeds 10% of total portfolio weight.
3. WHERE a User's Investment_Horizon is greater than 12 months and the Portfolio_Recommendation contains Bond allocations, THE Risk_Engine SHALL propose at least one interest rate Swap to manage duration risk.
4. WHERE a User's risk tolerance is set to aggressive, THE Risk_Engine SHALL include at least one Exotic_Option strategy in the Derivative_Overlay.
5. THE Risk_Engine SHALL calculate and display the estimated cost of each proposed derivative position as a percentage of the total portfolio value.
6. THE Risk_Engine SHALL calculate and display the maximum potential loss reduction (in percentage points of portfolio value) for each proposed derivative position.
7. IF the Market_Data_Service cannot provide pricing data for a required derivative, THEN THE Risk_Engine SHALL exclude that derivative from the Derivative_Overlay and include a descriptive notice in the response.
8. THE Risk_Engine SHALL use Black-Scholes pricing for Vanilla_Option valuations and SHALL document the inputs (spot price, strike, volatility, risk-free rate, time to expiry) in the Derivative_Overlay response.

### Requirement 5: Market Data Ingestion and Distribution

**User Story:** As a retail investor, I want the portal to display up-to-date market prices and data, so that my portfolio recommendations and risk assessments reflect current market conditions.

#### Acceptance Criteria

1. THE Market_Data_Service SHALL fetch real-time or near-real-time price quotes for all Instruments referenced in active Portfolio_Recommendations from the configured Market_Data_Provider.
2. WHEN real-time streaming data is available from the Market_Data_Provider at no cost, THE Market_Data_Service SHALL establish a WebSocket_Feed and push price updates to connected frontend clients with a latency of no more than 5 seconds from the provider update.
3. WHEN real-time streaming is not available at no cost, THE Market_Data_Service SHALL poll the Market_Data_Provider at intervals of no more than 60 seconds and update the Data_Cache.
4. THE Market_Data_Service SHALL use Yahoo Finance (via the public yfinance-compatible REST endpoint) as the primary Market_Data_Provider and Alpha Vantage free tier as the fallback provider.
5. IF the primary Market_Data_Provider is unavailable for more than 30 seconds, THEN THE Market_Data_Service SHALL switch to the fallback Market_Data_Provider and log a warning.
6. IF both Market_Data_Providers are unavailable, THEN THE Market_Data_Service SHALL serve the most recent Data_Cache snapshot and include a staleness timestamp in all market data responses.
7. THE Market_Data_Service SHALL cache the most recent price quote for each Instrument in the Data_Cache and SHALL refresh the cache on each successful provider poll or push update.
8. WHEN a frontend client requests historical price data for an Instrument, THE Market_Data_Service SHALL return daily OHLCV (open, high, low, close, volume) data for the requested date range, up to a maximum of 5 years.
9. THE Market_Data_Service SHALL support retrieval of reference data (instrument name, ISIN or ticker, currency, exchange) for all Instruments in the Portal's supported universe.

### Requirement 6: Instrument Universe Management

**User Story:** As a retail investor, I want the portal to support a broad set of instruments across asset classes and regions, so that my portfolio can be properly diversified.

#### Acceptance Criteria

1. THE Portal SHALL support the following asset classes: Equity, Bond, Vanilla_Option, Swap, Exotic_Option, Structured_Product.
2. THE Portfolio_Engine SHALL maintain a curated universe of at least 50 Equities covering the Regional_Preferences: North America, Europe, Asia-Pacific.
3. THE Portfolio_Engine SHALL maintain a curated universe of at least 20 Bonds covering sovereign and investment-grade corporate issuers across the supported Regional_Preferences.
4. WHEN a User selects a Regional_Preference of North America, THE Portfolio_Engine SHALL include Instruments listed on NYSE, NASDAQ, or major US bond markets.
5. WHEN a User selects a Regional_Preference of Europe, THE Portfolio_Engine SHALL include Instruments listed on LSE, Euronext, or major European bond markets.
6. WHEN a User selects a Regional_Preference of Asia-Pacific, THE Portfolio_Engine SHALL include Instruments listed on TSE, ASX, or major Asia-Pacific bond markets.
7. WHERE a User selects Global as a Regional_Preference, THE Portfolio_Engine SHALL draw Instruments from all supported regional universes.
8. THE Portal SHALL store Instrument reference data (ticker, ISIN, name, asset class, exchange, currency) in the persistence layer and SHALL update it at least once per trading day.

---

### Requirement 7: Portfolio and Recommendation Dashboard

**User Story:** As a retail investor, I want a clear dashboard showing my current portfolio recommendation, derivative overlay, and live market data, so that I can monitor and act on my investments.

#### Acceptance Criteria

1. WHEN an authenticated User navigates to the dashboard, THE Portal SHALL display the most recent Portfolio_Recommendation and its associated Derivative_Overlay within 3 seconds of page load.
2. THE Portal SHALL display the current price, daily change (absolute and percentage), and currency for each Instrument in the Portfolio_Recommendation.
3. WHEN the Market_Data_Service pushes a price update via WebSocket_Feed, THE Portal SHALL update the displayed price for the affected Instrument within 1 second of receiving the update.
4. THE Portal SHALL display a visual allocation chart (e.g. pie or donut chart) showing the percentage weight of each Instrument in the Portfolio_Recommendation.
5. THE Portal SHALL display the estimated portfolio expected return and estimated portfolio volatility based on current market data.
6. THE Portal SHALL display each proposed derivative position from the Derivative_Overlay with its estimated cost and maximum potential loss reduction.
7. WHEN an authenticated User selects a historical Portfolio_Recommendation from the history list, THE Portal SHALL display that recommendation's details and the market prices at the time of generation.
8. THE Portal SHALL display a staleness warning banner when the market data displayed is older than 120 seconds.

---

### Requirement 8: Interactive Financial Profile Wizard

**User Story:** As a retail investor, I want a guided step-by-step wizard to enter my financial profile, so that I can provide accurate information without feeling overwhelmed.

#### Acceptance Criteria

1. THE Portal SHALL present the Financial_Profile collection as a multi-step wizard with no more than 5 steps.
2. WHEN a User completes a wizard step and clicks "Next", THE Portal SHALL validate the inputs for that step and SHALL NOT advance to the next step if any input is invalid.
3. IF a User enters an invalid value in a wizard field, THEN THE Portal SHALL display an inline validation message adjacent to the invalid field within 200 milliseconds.
4. WHEN a User clicks "Back" in the wizard, THE Portal SHALL return to the previous step and SHALL retain all previously entered values.
5. WHEN a User completes all wizard steps and submits the Financial_Profile, THE Portal SHALL display a confirmation message and navigate to the dashboard within 3 seconds of a successful Profile_Service response.
6. THE Portal SHALL display a progress indicator showing the User's current step and total number of steps throughout the wizard.
7. WHEN a User returns to the wizard with an existing Financial_Profile, THE Portal SHALL pre-populate all wizard fields with the values from the most recently persisted Financial_Profile.

---

### Requirement 9: Market Data Serialization and Parsing

**User Story:** As a developer, I want market data to be reliably serialised and deserialised between the backend and frontend, so that price data is never corrupted or lost in transit.

#### Acceptance Criteria

1. WHEN the Market_Data_Service serialises a market data snapshot to JSON, THE Market_Data_Service SHALL produce a JSON object conforming to the defined MarketDataSnapshot schema (ticker, price, currency, timestamp, source, isStale).
2. WHEN the REST_API deserialises a JSON market data payload received from the Market_Data_Provider, THE REST_API SHALL parse it into a MarketDataSnapshot domain object.
3. IF a JSON market data payload is missing a required field (ticker, price, or timestamp), THEN THE REST_API SHALL reject the payload with a descriptive parse error and SHALL NOT persist a partial snapshot.
4. THE Market_Data_Service SHALL implement a pretty-printer that formats MarketDataSnapshot objects back into valid JSON conforming to the MarketDataSnapshot schema.
5. FOR ALL valid MarketDataSnapshot objects, serialising then deserialising then serialising SHALL produce an identical JSON representation (round-trip property).

---

### Requirement 10: System Reliability and Non-Functional Requirements

**User Story:** As a retail investor, I want the portal to be reliable, responsive, and secure, so that I can trust it with my financial data and decisions.

#### Acceptance Criteria

1. THE Portal SHALL return HTTP responses for all REST_API endpoints within 5 seconds under normal load (up to 100 concurrent users).
2. THE Portal SHALL be deployable via Docker Compose using the existing Dockerfile.frontend and backend/Dockerfile conventions.
3. THE Auth_Service SHALL enforce HTTPS for all communications between the frontend and backend in production deployments.
4. THE Portal SHALL sanitise all User-supplied input on the backend before persisting or processing it, to prevent SQL injection and XSS attacks.
5. WHILE a User session is active, THE Portal SHALL automatically refresh the JWT using the refresh token before the JWT expires, without interrupting the User's workflow.
6. THE Portal SHALL log all authentication events (registration, login, logout, token refresh, failed login) with a timestamp and a non-PII user identifier.
7. IF the Portfolio_Engine or Risk_Engine encounters an unhandled exception during recommendation generation, THEN THE Portal SHALL return HTTP status 500 with a generic error message and SHALL log the full exception details server-side.
8. THE Portal SHALL support horizontal scaling of the backend services via stateless JWT authentication, with no server-side session state required.
